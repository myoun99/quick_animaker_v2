이 문서는 오디오 싱크 정비 + 멀티플랫폼 네이티브 배선의 **단일 기준점**이다.

## 0. 왜 하는가 (문제 정의)

**미리보기가 거짓말을 하면 작화 데이터가 오염된다.**

유저 사례(TVPaint, 기타 연주 신): 립싱크와 작화가 3초 컷에서 3콤마쯤 어긋났다. 재생이 밀리면 작화자는 **밀린 미리보기에 맞춰 그리고**, 그 그림은 최종 출력에서 반대 방향으로 어긋난다. 즉 이것은 재생 품질 문제가 아니라 **정확성 문제**다.

수치 확인: 23.976 vs 24 오차는 0.1% = 3초(72프레임)에서 **0.072프레임**. 3프레임은 그 40배이므로 **풀다운 오차가 아니다** — 재생 엔진이 그림과 소리를 락하지 않은 결과다.

### 현재 코드의 실제 결함 (2026-07-20 확인)

- `lib/src/ui/playback/audio_playback_sync.dart` `_startClip`: 클립 시작 시점에 `startAt(seek+resume)`을 **`unawaited`로 던지고 클럭은 기다리지 않는다.** 플랫폼 플레이어의 시작 지연(50~200ms = 24fps에서 1.2~4.8프레임)이 **보정 없이** 실린다. 컷마다 클립을 재시작하므로 매번 반복된다.
- `_clampToFileLength` / `AudioPeaks.durationFrames`: 초→프레임 변환에 `.ceil()` — 항상 + 방향으로 최대 1프레임.
- 재생 위치는 자유 진행 타이머(`elapsedToGlobalFrame`)에서 나온다 → 오디오 장치 클럭과 **락이 없다**.

## 1. 목표 아키텍처 (프리미어/리졸브/프로툴즈와 동일)

| 원칙 | 내용 |
|---|---|
| **오디오가 마스터 클럭** | 재생 위치 = **장치에 전달된 샘플 수**. 자유 타이머 금지 → 누적 드리프트 구조적으로 0 |
| **클립을 "재생"하지 않고 믹스를 만든다** | 콜백마다 각 클립에서 `[S, S+N)` 샘플을 직접 퍼와 게인·페이드 적용 후 합산. **"시작 지연" 개념 자체가 소멸**, 클립 시작 정확도 ±1샘플 |
| **그림이 소리를 따라간다** | 매 표시마다 장치의 실제 재생 샘플을 질의 → 프레임 인덱스로 환산 → 그 프레임 표시. 늦으면 **그림을 버린다.** 소리는 절대 기다리지 않는다 |
| **출력 지연 보정** | 장치가 보고하는 지연만큼 표시를 당김. 잔차는 사용자 A/V 오프셋 설정으로 |

### 보장 범위 (정직하게)

- **보장**: 어떤 재생 길이에서도 소리↔그림 **누적 오차 0**. 클립 간 ±1샘플. 부하 시 그림만 손실.
- **보장 불가(물리)**: 화면 파이프라인 + 오디오 장치의 **절대 지연**. 장치 보고값으로 보정 후 잔차는 통상 1프레임 미만, 남은 상수는 사용자 슬라이더로 제거.

### 협상 불가 규칙

1. **오디오 콜백 안에서 Dart 호출 금지.** 실시간 스레드에 GC 언어 진입 불가. Dart는 클립 업로드·트랜스포트 제어·위치 폴링만. (기존 래스터 엔진의 "Dart가 준비하고 C가 돈다"와 동일 규율)
2. **디코드·리샘플은 임포트 시 1회**, 프로젝트 샘플레이트(기본 48kHz)로 통일해 PCM 보관. 콜백은 memcpy + 가중합만.
3. **믹서는 순수 함수**(클립 목록 + 샘플 범위 → 버퍼) → **하드웨어 없이 바이트 단위 단위 테스트**. Dart 레퍼런스 ↔ C 패리티는 기존 엔진 방식 그대로.
4. **안전망**: 장치 열기 실패 시 기존 플랫폼 플레이어 경로로 강등. **어떤 경우에도 무음 금지.**

### 오디오 데이터 모델 규칙 (유저 확인 완료)

사운드는 **글로벌 트랙 소유**이고 컷은 그 창(window)을 표시할 뿐이다. 이 구조가 옳다 — 컷을 넘는 대사·음악이 성립하고, **컷 경계에서 재시작이 없어** 컷마다 시작 지연이 실리는 사고(TVP 증상)가 구조적으로 불가능해진다. 따르는 규칙:

- **믹서는 GLOBAL 폼만 읽는다.** 컷 로컬 디스플레이 클론(`trackSeWindow.displayLayer`)은 표시 전용 — 먹이면 컷 밖으로 넘치는 클립이 잘린다.
- **프레임→샘플은 64비트 정수 연산**: `sample = frame × sampleRate × den ÷ num`. float 누적 금지.
- **컷 재정렬·길이 변경 = 스케줄 무효화 후 재생성**(클립은 셀 `frameId`에 앵커돼 타이밍을 자동 추종).
- 한 컷만 재생해도 클립 위치는 글로벌 그대로, 트랜스포트 시작 샘플만 그 컷의 글로벌 시작으로.

## 3. 2A — 멀티플랫폼 네이티브 배선

### 현 상태 (2026-07-20 코드 확인)

`native/CMakeLists.txt`는 이식성 있게 작성돼 있으나 **실제로 빌드·번들하는 것은 `windows/CMakeLists.txt` 하나뿐**이다. Linux는 배선 없음, macOS·iOS·Android는 전무. Dart 로더(`qa_native_engine.dart:_tryOpen`)는 `.so`/`.dylib`를 시도하지만 그 파일을 만드는 주체가 없다 → **네이티브 엔진은 실질적으로 Windows 전용**이며 타 플랫폼에서는 조용히 스탠드다운한다(그래서 지금까지 드러나지 않았다).

**오디오는 스탠드다운이 불가능**하므로(무음 앱은 성립하지 않음) 이 부채를 여기서 갚는다.

### 해법: 로컬 FFI 플러그인

```
packages/qa_native/
  src/                    ← C 소스 (native/ 에서 이전)
  ios/qa_native.podspec        → CocoaPods 컴파일, 앱에 정적 링크
  macos/qa_native.podspec
  android/build.gradle         → externalNativeBuild(CMake) → jniLibs
  linux/CMakeLists.txt
  windows/CMakeLists.txt
```

**Xcode 프로젝트(pbxproj)는 절대 손대지 않는다:**
- UUID 덩어리라 diff 리뷰 불가·머지 충돌 지옥, `ios/` 재생성 시 소실
- iOS/macOS를 각각 두 번 해야 하고 **안드로이드는 Xcode로 불가능** → 메커니즘이 3개로 분열
- **결정적**: iOS는 앱 번들에서 독립 `.dylib` 로드가 불가. **정적 링크 + `DynamicLibrary.process()`** 가 유일한 지원 경로이며, FFI 플러그인 구조가 그것을 처리한다 → **현재 로더는 iOS에서 구조적으로 동작 불가**

**부수 효과(큼)**: 배선이 끝나면 아이패드·갤탭이 **타일 래스터라이저 + 플러드필 가속**을 함께 얻는다.

**위험 관리**: 현재 동작하는 Windows 빌드는 **기존 경로 유지** → 플러그인이 검증된 뒤 마지막에 이행한다.

### 플랫폼별 함정 (착수 전 파악 완료)

- **iOS/iPadOS**: `AVAudioSession` 카테고리(playback)·모드, 인터럽션(전화) 복귀, 라우트 변경(이어폰 착탈), 기기 주도 샘플레이트 변경. miniaudio가 세션을 전부 관리하지 않으므로 우리가 붙인다.
- **Android**: 오디오 포커스 상실/복귀, AAudio(API 26+) 저지연 버퍼, 구형 OpenSL ES 폴백, NDK ABI(arm64-v8a, x86_64).
- **macOS**: 출력 전용이면 샌드박스 추가 권한 불필요(마이크만 필요).
- **ARM Windows**: 동일 소스, cmake 타깃 추가.
- **웹**: 스탠드다운(목표 플랫폼 아님).

## 4. RT — 유리수 타임베이스

23.976 = **24000/1001**. double은 근사값이며 프로 앱은 전부 분수로 저장한다.

**작업이 작은 이유**: 23.976 프로젝트에서도 **프레임은 정수**다. 바뀌는 것은 "1프레임이 몇 초냐"뿐 → 그리드·6f선·시트 행·초 라벨은 **정수 카운팅 베이스(24)를 그대로 사용**하며 한 줄도 건드리지 않는다(프로 앱의 타임코드도 23.976에서 24프레임마다 1초로 센다).

분수가 필요한 지점은 4곳뿐(코드 확인):
1. `services/playback/playback_frame_mapping.dart` — 재생 시계
2. `ui/playback/audio_playback_sync.dart` — 클립 초↔프레임
3. `services/audio/audio_peaks_extractor.dart:durationFrames` — 파형 길이
4. `ui/export/video_export_service.dart` — ffmpeg `-r` (ffmpeg는 `24000/1001` 분수를 받는다)

설계: `ProjectFrameRate { numerator, denominator }` + `countingBase`(24, 30). 프리셋이 셋을 함께 보유. 현행 int `fps`는 카운팅 베이스로 승계.

### fps 변경 시맨틱 (유저 결정)

- **기본은 재해석(conform)**: 데이터 그대로, 한 장당 시간만 변경. 셀 복제/삭제(리타임)는 기본이 되어서는 안 된다.
- 오디오는 실제 초로 존재하므로 0.1% 어긋남 → **레이트 변경 시 "오디오 그대로 / 0.1% 당김" 선택만 묻는다.**
- 23.976 소재를 24 프로젝트에 넣기: **프레임 정확**(conform + 오디오 0.1% pull) 또는 **시간 정확**(41.7초마다 1프레임 중복) 중 택일 — 24 ≠ 24000/1001이므로 둘 다는 수학적으로 불가능.
- **샘플레이트: 48kHz 기본 + 프로젝트 설정 노출.**

## 5. CI

공개 저장소 전환 완료(2026-07-20)로 표준 러너 **비용 0**.

| 잡 | 시점 | 내용 |
|---|---|---|
| Linux | 매 푸시 | `flutter analyze` + 풀 테스트 + `native/` C 컴파일 + Android 빌드 |
| Windows | 매 푸시 | 네이티브 DLL 빌드 + 패리티 테스트 |
| **macOS** | **하루 1회 + 수동** | macOS 앱 빌드, **iOS 시뮬레이터 빌드**(서명 불필요), `native/` clang 컴파일 |
| TestFlight | iPad 도착 후 | fastlane 서명 → App Store Connect 업로드 |

### 검증 3층

1. **컴파일 검증** — 계정·기기 없이 오늘부터. 네이티브 C 이식성 문제의 대부분을 여기서 잡는다.
2. **iPad 실기 설치** — Apple Developer $99/년 + TestFlight. **Mac 불필요.** 가입은 **아이패드 주문일에**(승인 24~48h, $99 시계가 승인 시점부터).
3. **실기 계측** — 오디오 지연 실측은 iPad/갤탭에서만. 2D의 인스펙터가 이를 담당.

**정직한 한계**: macOS는 "컴파일 검증됨, 체험 미검증"으로 남는다(실행해 볼 사람이 없음).
